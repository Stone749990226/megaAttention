# megaAttention 执行方案（fused FA + O_proj + NVLS AR persistent kernel）

> 目标：严格按 `docs/design/causal_varlen_prefill_persistent_fa_oproj_ar_plan_zh.md`，
> 把 real FA、real O_proj、real NVLS AllReduce 收敛进同一个 persistent kernel
> （`src/mega_attention/kernels/sm90/fused_fa_oproj_ar.py`）。
> 范围不变：Hopper SM90、causal、varlen full prompt prefill、`q_len == k_len`。

- **真实**：grid_sync、FA atomic counter、O_proj ready queue（reserve/ordered-publish/CAS-pop）、

### 三个硬缺口
1. **multi_seq 死锁**（blocker）：`test_fused_fa_path.py` 的 `multi_seq=[200,64,300]` 在 fused
   scheduler 死锁；同样输入在 standalone `test_fa_packed.py` 通过 → FA payload 本身没问题，
   死锁由 fused 持久循环上下文引入。
2. **real O_proj 未接入 dispatcher**：`oproj_tile.py` 是单 CTA，需按设计文档 §1335/§1400 做
   O_proj warp specialization、复用长寿命 A/Wo pipeline state、与 FA mode 共享 tensor SMEM（§1483）。
3. **real NVLS AR 未接入、无端到端/多卡 fused 数值验证**：standalone AR 协议可用，但 fused 上下文
   的 FA→O_proj→AR 链从未数值校验。

## 1. 总体策略

- 单一主线：所有改动只服务于让 `fused_fa_oproj_ar.py` 跑通完整链。不做对外 API、不引入第一版范围外机制。
- 小步 + gate：每个阶段有明确「完成判据（gate）」，gate 不过不进入下一阶段。
- 先地基后广度：先修死锁解锁 FA 完整验证，再接 O_proj，再接 AR；AR 先单卡 `tp_size=1` 闭环，最后才上 8 卡。
- Hopper-only 结论：任何 kernel 正确性/死锁/性能结论必须在实际 Hopper 上跑出来才算数。
- 横切不变量集中维护（见 §6），每阶段动 SMEM / pipeline state / 内存序时回查设计文档对应章节。

阶段依赖：**P0（死锁）→ P1（O_proj 单卡）→ P2（AR 单卡闭环）→ P3（8 卡 NVLS + 性能）**。

环境前置（每次 GPU 测试前）：
```bash
export LD_LIBRARY_PATH=/usr/local/cuda-13.0/compat:$LD_LIBRARY_PATH
```

## P0 — 修复 fused FA 路径 multi_seq 死锁

**目标**：fused kernel 在 `multi_seq=[200,64,300]` 等多序列 varlen 输入下 FA 路径不死锁，
O_scratch 数值匹配参考，调度 exactly-once + ordered + 正常终止。这是后续一切的 gate。

**前置**：standalone `test_fa_packed.py` multi_seq 通过（已满足）；单序列 fused 用例通过（已满足）。

**诊断优先（systematic-debugging，先定位再改）**
死锁是 fused 持久上下文引入的。按可疑度排序的假设：
- **H1（最可疑）长寿命 pipeline state 跨 task 失配**：`fa_k_prod/cons`、`fa_v_prod/cons` 在
  dispatch loop 外创建、跨 task 推进。producer（WG0 warp0）与 consumer（WG1/WG2）每 task 各推进
  `nblk` 次，理论应平衡；但 multi_seq 下不同 task 的 `nblk` 差异 + 全掩码 WG2 组合可能破坏平衡。
  注意：`varlen_200`（多 nblk）已通过 → 单纯 nblk 变化不是根因，重点查与**全掩码 WG2**的组合。
- **H2 全掩码 WG2 的 pipeline 参与**：`seq=64 → valid_m=64`，WG2（rows 64..127）全掩码；`seq=300`
  末 tile `valid_m=44` 同样全掩码。需确认全掩码 WG2 仍**无条件**走完所有 `consumer_wait/release`
  与 `wait_group`，否则它与 producer/另一 consumer 失配。
- **H3 ordered-publish 自旋活锁**：`publish_oproj`（`:160-181`）在 `tidx==0` 内自旋等
  `publish_tail==start`；与 `schedule_pick` leader-only 选择 + 全 CTA `sync_threads` 交互时，
  leader 自旋会让整个 CTA 卡住（其余 WG 已在 `:754` 等 leader）。需验证 reservation 序是否严格
  == 完成序（理论上是，但要确认 multi_seq 下无并发竞态破坏）。
- **H4 终止条件竞态**：`schedule_pick` 读三计数器判 DONE 与 in-flight publish/claim 的可见性顺序。

**诊断步骤**
1. 用 `scratch/_probe_multiseq.py` 确认死锁发生在 execute（已有探针：COMPILE_DONE / LAUNCH_RETURNED / SYNC_DONE）。
2. 二分隔离：
   - 强制 `num_ctas=1`，看是否仍死锁（区分「跨 CTA 调度」vs「单 CTA 内 pipeline」）。
   - 临时让 leader 只选 FA、禁用 O_proj/AR stub 选取（区分「mode 交错」vs「纯 FA 持久循环」）。
   - 构造最小复现：只留 `[64]`（单全掩码序列）、`[200,64]`、`[64,300]`，定位是哪种组合触发。
3. 据复现集合锁定 H1/H2/H3 之一，给出最小修复（优先改 payload 的 pipeline 收尾/drain 对齐，
   或全掩码 WG2 的无条件 pipeline 参与，避免动调度协议）。

**改动文件**：主要 `kernels/sm90/fused_fa_oproj_ar.py`（FA payload 段 `:608-744` 的 pipeline
收尾 / 全掩码分支）；必要时 `tests/fused/test_fused_fa_path.py`（补最小复现用例）。

**验证命令**
```bash
python tests/fused/test_fused_fa_path.py        # multi_seq 不再死锁，4 用例全过
python tests/fused/test_fa_packed.py            # 回归 standalone FA 仍过
pytest tests/fused/test_scheduler_skeleton.py   # 调度协议回归
```

**Gate**：`test_fused_fa_path.py` 全部用例（含 multi_seq）通过；O_scratch 数值匹配参考
（误差 < 2e-2）；调度 `order_err==0`、exactly-once、正常 SYNC 返回；standalone 回归不退化。

**风险/回退**：若根因在长寿命 state 设计层面（而非实现 bug），需回设计文档 §「长寿命 pipeline
state / drain」核对，并停下来与用户确认是否调整 drain 规则——不擅自改 invariant。

## P1 — 接入 real O_proj（单卡 tp_size=1，AR 仍 identity）

**目标**：dispatcher 的 MODE_OPROJ 调用真实 GEMM，从 O_scratch row tile 读入、`@ W_o_local`
写出本 rank partial，到 C_sym workspace；fused 上下文数值匹配 `oproj_reference`。AR 暂保持 identity。

**前置**：P0 gate 通过。

**关键设计对照（动手前必读）**
- §493 O_proj task identity、§428 规模估算（A = O_scratch 完整 128 行 FA tile，B = W_o_local）。
- §1335/§1400 O_proj/AR warp specialization 与 pipeline（WG0=DMA、WG1/WG2=MMA，K-chunk stage 复用）。
- §1483 Shared Memory 预算：O_proj mode 与 FA mode **复用同一块 tensor SMEM**，需核对 A/Wo 占用
  不超过 FA 段已分配的 sQ/sK/sV 总量；mbarrier 复用规则同 §「长寿命 pipeline state」。
- §1053 长寿命 `oproj_ab_prod/cons` 在 dispatch loop 外创建、跨 task 推进（不可 per-task 重建）。

**实施步骤**
1. 把 `oproj_tile.py` 的 OProjTile 微核逻辑（A K-major、B = `transpose_view(W_o)` MN-major、
   共享 A/B pipeline、predicated store、super_group N-tile 循环）抽成可内联的 payload 函数。
2. 在 fused kernel 增加 O_proj 的 SMEM（A/Wo stage）与 mbarrier，纳入 `Smem` struct；核对 §1483 预算，
   确认与 FA 段 SMEM 复用/不超预算（必要时调 stage 数）。
3. dispatch loop 外创建 `oproj_ab_prod/cons` 长寿命 state；MODE_OPROJ 分支替换 stub 为真实 payload，
   消费 `decode_oproj_slot` 解出的 `(row_tile, n_super_group, base_out_n_tile, valid_n_tiles)`。
4. partial 写到 C_sym workspace（host 侧用 `csym_numel()` 分配）；保留现有 `ready_count_owner` /
   `ar_probe` 推进逻辑（tp_size=1 即时 last-arriver）。
5. host 侧测试装载 W_o、C_sym、O_scratch，串 FA→O_proj。

**改动文件**：`kernels/sm90/fused_fa_oproj_ar.py`（SMEM/mbarrier/O_proj payload/dispatch）；
新增 `tests/fused/test_fused_oproj_path.py`（FA+O_proj 端到端，AR identity）；
参考 `reference/fused.py:oproj_reference`。

**验证命令**
```bash
python tests/fused/test_fused_oproj_path.py     # FA+O_proj partial 匹配 oproj_reference
python tests/fused/test_fused_fa_path.py        # P0 回归不退化
pytest tests/kernels/test_oproj_tile.py         # microkernel 回归
```

**Gate**：fused C_sym partial 对 `oproj_reference`（fp32 shadow）误差 < 3e-2；
含 multi_seq + valid_n tail + ragged super_group 形状；FA 路径回归不退化；正常终止。

**风险/回退**：SMEM 预算超限 → 调 O_proj stage 数或 N_TILE；若 FA/O_proj SMEM 复用与 mbarrier
复用冲突无法两全，停下来回 §1483 + §「长寿命 pipeline state」与用户确认。

## P2 — 接入 real NVLS AR（单卡 tp_size=1 闭环）

**目标**：MODE_AR 调用真实 multimem 路径（单卡 `tp_size=1` 下退化为 local copy/identity-on-real-data，
但走真实 `multimem.ld_reduce/st` 代码路径与内存序），打通整条 FA→O_proj→AR 数据流与数值闭环。

**前置**：P1 gate 通过。

**关键设计对照**：§1718 O_proj/AR 尾声、§1734 跨 rank ready 方式、§1821 非阻塞 AR owner reduce task、
§1883 NVLS AllReduce 执行方式、§1992 workspace 生命周期与全局同步、§2167 local grid_sync / §2204 nvl_barrier。

**实施步骤**
1. 从 `oproj_ar.py` 抽取 NVLS helper（`multimem_ld_reduce_8xbf16`、`multimem_st_4xb32`、
   release/acquire 内存序、per-batch flag 思路），适配 fused 的 AR owner task 粒度
   （`ar_slot_id = row_tile*num_super_groups + n_super_group`）。
2. host 侧用 symmetric memory 分配 C_sym + flag，构造 multicast 指针（`tp_size=1` 也走同一路径，
   world_size=1 时 reduce 退化但代码路径真实）。
3. MODE_AR 分支替换 identity stub：claim AR owner → `multimem.ld_reduce`+`multimem.st` →
   等 final store 完成 → `ar_done_count++`，遵守 §971 done counter 递增点。
4. 校验整条链：FA→O_proj→AR 结果对 `full_chain_reference`（tp_size=1 下 AR 为恒等）。

**改动文件**：`kernels/sm90/fused_fa_oproj_ar.py`（AR payload + host workspace/flag）；
新增 `tests/fused/test_fused_full_chain.py`（tp_size=1 端到端）；参考 `reference/fused.py:full_chain_reference`。

**验证命令**
```bash
python tests/fused/test_fused_full_chain.py     # FA→O_proj→AR(tp=1) 对 full_chain_reference
python tests/fused/test_fused_oproj_path.py     # P1 回归
```

**Gate**：tp_size=1 整链数值匹配 `full_chain_reference`（误差 < 3e-2）；AR exactly-once /
terminal-protected；走真实 multimem 代码路径；正常终止。

**风险/回退**：单卡 multimem 语义/对称内存初始化与多卡不一致 → 在 §1883 / §2204 核对；
若 tp_size=1 无法真实触发 NVLS 路径（必须 ≥2 卡），降级为「local reduce 验证数据流 + P3 直接上多卡验证 NVLS」，并记录该取舍。

## 6. 横切不变量（每阶段动到就回查设计文档）

- **长寿命 pipeline state**（§991-1071）：mbarrier kernel 启动初始化一次、跨 task 复用；
  `PipelineState` 软件游标在 dispatch loop 外创建、payload 内推进；**禁止 per-task 重建 state**；
  task 结束 drain（非 reset）。
- **mode 切换 drain**（§978-989）：切 mode 前 TMA 完成、WGMMA wait 完、当前 mode mbarrier/stage 收尾、
  所有 warp 回到一致同步点。
- **SMEM 预算**（§1483-1717）：FA mode 与 O_proj mode 复用同一块 tensor SMEM；动 stage/tile 前核对预算。
- **内存序 / happens-before**（§953-976, §1290）：done counter 递增点必须晚于对应数据 store 完成 +
  ready/publish 发布；release/acquire 与 sys-scope fence 保持设计文档要求。
- **AR owner 协议**（§908-941）：`ar_owner_probe_bits` 是「已 ready 可执行」而非「某 rank 刚完成」；
  owner claim 用 CAS，single-owner。

## 8. 第一版仍不做（§2265）

decode、append/chunked prefill、`seqlen_q != seqlen_k` 尾部对齐、paged KV、SplitKV、
partial O/LSE combine、非 causal、非 SM90、通用 public API。如需引入须先与用户确认。

## 9. 需用户拍板的点（出现时停下来）

- P1 若 FA/O_proj SMEM 复用与 mbarrier 复用无法两全，方案取舍前先确认。
