megaAttention 独立 fused FA + O_proj + NVLS 执行计划
Summary
目标是在 megaAttention/ 内实现独立算子库原型，不接入 FlashAttention public API；FA4 V4 Hopper 只作为 SM90 TMA/WGMMA/pipeline/mask 参考。执行顺序采用“先协议、后数学、再融合、最后 NVLS”的分层路线，每一阶段都必须有可运行验证和停下来审阅点，避免全量一气呵成。
仓库当前处于"半迁移、导入即崩"状态，第一步必须先修复并统一为 FA/O_proj/AR 共享 128 行 row tile：
- row_desc.py 仍是旧双 tile 模型（build_fused_meta/FusedMeta，OPROJ_M_TILE=64），能 import。
- tests/fused/test_scheduler_skeleton.py 与 reference/fused.py 已写成新统一 API（import build_row_desc / RowDescMeta），但 row_desc.py 没这两个符号，二者当前 ImportError 跑不起来。
- 因此 Step 1 不只是"重构"，而是同时修复这个破损。注意：记忆中"Phase 0+1 done & validated"是在另一版 row_desc 下做的，开工前需重新确认 Phase 0 是否仍为绿。
Key Changes
独立库落点：megaAttention/。
不接入 flash_attn/cute/interface.py，不修改 flash_attn_varlen_func。
主线验证环境：Hopper/NVLS 服务器
每个 AI 任务包必须遵守：先读 causal_varlen_prefill_persistent_fa_oproj_ar_plan_zh.md 对应章节。
若实现发现设计与代码/硬件语义冲突，停止并提交问题说明，不继续“顺手优化”。
每阶段只引入一个新风险面：metadata、scheduler、FA math、O_proj math、跨 rank AR、最终融合分开推进。

Implementation Order
设计-代码契约对齐
更新 src/mega_attention/metadata/row_desc.py 的 host metadata：删除 64 行 O_proj subtile 模型，统一为 fa_row_tile_id、FA_M_TILE=OPROJ_M_TILE=128、slot_id = fa_row_tile_id * num_super_groups + n_super_group。
验收标准（硬性）：新 row_desc 必须以 build_row_desc / RowDescMeta 命名导出（不是 build_fused_meta/FusedMeta），且提供 .num_row_tiles / .M_TILE / .num_batch / .cu_seqlens_q / .cu_seqlens_k / .valid_m(t) / .q_tile_start(t)；目标是先让 tests/fused/test_scheduler_skeleton.py 与 reference/fused.py 的 import 通过（修复当前 ImportError），再谈 128 统一。顺手更新 row_desc.py 顶部对已删除的 设计文稿.md 的引用。
更新 test_row_desc.py，覆盖 varlen tail、q_len == k_len 前置条件、C_sym[fa_row_tile_id,m,out_n_tile,n] tile-padded 容量估算。
停下来审阅点：如果仍需要 64 行 O_proj tile，必须回到设计稿讨论，因为这会推翻当前 128 行统一 row tile 设计。

纯协议 scheduler skeleton
修正 src/mega_attention/kernels/sm90/fused_fa_oproj_ar.py 和 tests/fused/test_scheduler_skeleton.py，只做 stub payload，不做真实 FA/O_proj math。
验证内容：FA task exactly-once、head_ready_count 后才发布 O_proj queue、ordered publish 无洞、O_proj task exactly-once、AR owner task exactly-once、done counter 能退出。
暂时单 rank、device-scope atomic；不要引入 NVLS。
停下来审阅点：如果 ready queue 出现 head-of-line blocking 或死锁，先讨论是否保留 queue 还是回到 bitset/frontier。

单 tile FA microkernel
基于仓库中已在飞的 src/mega_attention/kernels/sm90/fa_ws.py 续作（git 显示 fa_tile_sm90.py / test_fa_tile.py 已被删除，真正在改的是 fa_ws.py，其 step3 目标 = tile_m=128 + WG1/WG2 各 64 行 + causal + varlen，与本步一致）。开工前明确二选一：续作 fa_ws.py，或显式废弃它重建 fa_tile_sm90.py，不要并行造两份。注意 fa_ws.py 依赖外部 from quack import layout_utils，需确认该依赖可用。
独立验证 Hopper FA tile，不接 scheduler。
顺序推进：QK WGMMA -> causal mask -> online softmax -> PV -> varlen valid_m/k_len 谓词。
第一版可用同步 copy 或保守 pipeline；TMA/WG overlap 不在此阶段强求。
对照 torch reference，默认只覆盖 causal varlen prefill 且 q_len == k_len。
停下来审阅点：若 causal mask 需要 seqlen_k - seqlen_q offset，说明场景已越过完整 prompt prefill，需要回到设计范围。

单 tile O_proj microkernel
以 oproj_ar.py 的 WGMMA GEMM 代码为参考，先实现无 NVLS 的 O_scratch row tile @ W_o_local -> partial C_sym tile。
输入布局固定为 O_scratch[fa_row_tile_id,128,H_local,D]，GEMM K 维为 H_local * D。
验证 valid_m、valid_n 谓词和 tile-padded C_sym offset。
停下来审阅点：如果 shared A/B stage 的双 WG release 协议复杂度过高，先降级成保守 wait_group(0) 后 release，不做 overlap 优化。

FA -> O_proj 单 rank融合
把真实 FA payload 替换 scheduler stub 的 do_fa，写 O_scratch_local。
把真实 O_proj payload 替换 do_oproj，读完整 128 行 FA tile 的所有 local heads。
AR 仍保持单 rank identity/stub，先验证端到端 FA + O_proj 数值正确。
停下来审阅点：若 mode 切换需要 shared memory layout 重构，先停下来确认 overlay 方案，不在同一任务里同时改 scheduler 和 math。

单 rank persistent 完整路径稳定化
加入 init/exit grid_sync、control state 清零、persistent CTA 数量参数化。
硬约束：num_persistent_ctas 必须 clamp 到实际能同时常驻的 CTA 数（occupancy）。原因有两个死锁通道——(1) grid_sync 要求所有 CTA 都参与，多出的 CTA 上不了 SM 就永远到不了 barrier；(2) publish_oproj 的 ordered-publish 自旋依赖别的 producer 推进 publish_tail，若超额 launch 导致某 producer 上不了 SM，前面 reserve 过的 producer 会全部 head-of-line 卡死。参数化时必须按 occupancy 上限封顶，并加用例验证超额 launch 不会卡。
验证不同 batch/seqlen/head/hidden/super_group 组合都能退出且数值正确。
此阶段仍不引入跨 rank symmetric memory。
停下来审阅点：任何 barrier 死锁、done counter 不一致、CTA 提前 return 都必须先写复现和协议分析。

NVLS control path 最小接入
只接入跨 rank control：symmetric ready_count_owner、ar_owner_probe_bits、ar_done_bits、init/exit nvl_barrier。
O_proj partial 可以先写可识别 checksum，不做真实 multimem reduce。
必须实现并验证：partial store completion -> fence_proxy("alias") -> system-scope release atomic。
注意一处设计内在不一致需在本步顺带量测：设计稿为 O_proj ready 发现否决了 bitset（方案A）改用 queue（方案B），理由是 consumer 轮询大量空 ready word 浪费 L2/调度；但 AR owner 发现仍用 bitset（ar_owner_probe_bits，skeleton try_claim_ar 是最坏 O(total_oproj) 的 cursor 扫描）。本步请量测大 hidden（probe 数到 256/512）下 AR probe 的空轮询占比。若明显成为瓶颈，记录数据并讨论是否给 AR 也换 queue；correctness 阶段不擅自改协议。
停下来审阅点：若 Python/CuTe DSL 对 system-scope atomic 或 multicast pointer 暴露不足，暂停讨论是否改用现有 oproj_ar.py 的通信结构。

NVLS reduce/store 接入
把 AR owner stub 替换为 multimem.ld_reduce.add + multimem.st，按 C_sym[fa_row_tile_id,m,out_n_tile,n] 同 offset in-place 写回。
第一版采用单 owner reduce 整个 ar_tile，不做 M-shard reduce。
对照 torch all_reduce reference 验证全 rank 输出一致。
停下来审阅点：若单 owner 成为明显瓶颈，只记录 profiling，不在 correctness 阶段改 M-shard 协议。

最终 fused kernel 收敛
合并真实 FA、O_proj、NVLS AR 到一个 persistent kernel。
开启小规模性能 profiling：先看正确性、死锁、workspace、SMEM 占用，再看 overlap。
只在 correctness 稳定后调整 super_group_n_tiles、CTA preference table、pipeline stage 数。

Assumptions
第一版只支持 Hopper SM90、causal varlen 完整 prompt prefill、q_len == k_len。
不支持 decode、append/chunked prefill、paged KV、SplitKV、block sparsity、PackGQA 优化。
megaAttention/ 是独立算子库原型目录；FA4 代码只参考，不作为 public API 集成目标。
默认 ROW_M_TILE=FA_M_TILE=OPROJ_M_TILE=AR_M_TILE=128，N_TILE=128，super_group_n_tiles=2 或 4，先从 4 开始。
遇到设计冲突、硬件语义不确定、通信协议需要重做、或性能优化会改变同步协议时，执行必须停止并与你讨论。
